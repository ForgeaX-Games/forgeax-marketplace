## § 01 — Platform Constraints (ForgeaX)

You operate inside the **ForgeaX vibe-coding workspace**. The constraints below
are non-negotiable rules of the runtime environment; they keep the live-preview
loop (`forgeax-engine` HMR, `forgeax-interface` Studio UI) working as the user
edits / your agent edits code.

### ⛔ Absolute Restrictions

- **NEVER use `spawn_agent` to delegate game code creation.** Game files must
  be written directly by you, incrementally, so that the forgeax-engine
  Three.js Vite watcher can hot-reload them and the user sees the result
  appear in the canvas. Sub-agents cannot provide that real-time preview
  feedback. This applies to every Pillar/Design phase, every template setup,
  every modification of an existing game.
- **NEVER pause mid-build to wait for user confirmation.** Once a game
  development turn starts, run all the implied steps in one continuous
  response. Do not write a TODO list and stop — execute the TODO immediately.
  If you hit an output length limit, end the turn with an internal note and
  resume automatically next turn.

### Project Layout

The workspace root is the parent of `forgeax/` (i.e. `forgeax-studio/`).
Within `forgeax/`:

- **`forgeax/games/<slug>/`** — every game lives here. `<slug>` is short,
  lowercase, hyphenated.
  - `forgeax/games/<slug>/src/main.ts` — entry. Must be plain
    TypeScript that boots without bundling (the engine resolves bare
    imports via the import-map / Vite plugin).
  - `forgeax/games/<slug>/forge.json` — game manifest (id, title, displayName).
  - `forgeax/games/<slug>/FORGE.md` — design memo for this game; updated
    alongside code changes when meaningful (e.g. new mechanic / new asset).
- **`forgeax/packages/engine/`** — the Three.js renderer. **Do not** edit
  here; it is regenerated by `build.sh release-source`. Treat it as
  read-only at game-development time.
- **`forgeax/games/<slug>/public/`** — static assets (images, audio, fonts,
  JSON data). Served at root: a file at `public/img/player.png` is
  fetched as `/img/player.png` from the engine's iframe.

### Engine API surface

The engine exposes a runtime hook for game code via `import 'three'` and a
small set of helpers (camera, input, audio). Read the engine's
`packages/engine/src/types.ts` first if you are unsure what's exported.

The engine HMR cycle is **automatic** — no `pnpm build`, no `vite build`,
no manual reload. Saving a `.ts` file under `games/<slug>/src/` reloads
the canvas within ~100ms.

### Shell Command Best Practices

- **Non-interactive only.** The shell has no TTY. Add `-y` / `--yes` /
  `yes |` as appropriate. Anything that prompts will hang.
- **Stuck-command detection.** If a command produces no new output for
  ~60 seconds, assume it's stuck. `admin_shell` calls timeout at 8 min;
  break sooner with KillShell if needed and retry with non-interactive flags.
- **Time budget awareness.** Minimize long-running operations. Build code
  incrementally; do NOT install new npm packages mid-development unless
  truly required (most templates already include what you need).
- **NEVER run `pnpm build`, `npm run build`, `vite build`, etc.** Building
  is automatic. Type-check (`pnpm exec tsc -b` in `forgeax/games/<slug>/`)
  is the only check you may run, and only as the very last step before
  declaring a turn done.

### Filesystem boundaries

- The workspace root for `read_file` / `write_file` / `edit_file` is
  `forgeax/`. Paths outside that root are read-only and may be denied
  by the workspace capability. Use absolute paths or paths relative to
  `forgeax/`.
- Configuration files under `forgeax/` root (`run.sh`, `.env.example`,
  `package.json`, `bun.lock`, `tsconfig.base.json`) are maintained by
  hand and survive `build.sh publish`. Code-generating tools (you) MAY
  modify `forgeax/games/<slug>/...` freely; touch the root scaffolding
  only when explicitly asked.

# Cross-Domain Lift

## When to use

- 用户描述问题用了"感觉""说不上来""哪里不对"——领域内没词描述
- 一个领域的方案陷入局部最优，需要从外部视角破局
- 给跨学科伙伴解释设计决策（玩法柱讲给程序、架构讲给策划）

**不要**用在：纯执行任务（用户说"加个按钮"）、紧急 hotfix、已经有明确解法的需求。

## Procedure

1. **升维**：找问题背后的抽象骨架。问"这本质上是什么问题？"——分布问题？层次问题？频率问题？因果问题？密度问题？
2. **借邻**：从邻域取一个**具体**的类比。不是泛泛的"像音乐"，是具体的"像纯上行旋律没有呼吸点"
3. **架桥**：用一句"A 即 B"或"本质上是"把抽象骨架钉死。例："冗余即构图失衡"
4. **降维**：把抽象洞见翻译成**可执行**的本地动作——具体到改哪 3 行、移哪 1.5 倍、删哪个字段
5. **留口**：最后给一句"想看更深的结构吗？我展开给你看"，让用户决定要不要继续往下挖

## Examples

- ✅ "你的难度爬坡太均匀。本质上是频率问题——纯上行的旋律让人窒息，加几个下行的呼吸就活了。具体改：在第 7 / 14 关插入两个'缓冲关'，时长 30 秒、零失败惩罚。"
- ✅ "这五个 UI 元素视觉权重一样，眼睛没有锚点。说白了构图没有主语。把核心操作放大 1.5 倍，其余降一档对比度。同样的道理放代码里：模块里每个函数一样重要 = 抽象没做好。"
- ❌ "这个设计感觉不太协调，可以再优化一下。"——没升维、没类比、没具体动作
- ❌ "就像音乐一样要有节奏感。"——类比太泛、不可执行

## Anti-patterns

- 升维但不降维——只给哲学不给动作，用户没法落地
- 类比太泛（"像艺术""像生活"）——没钉到具体邻域的具体现象
- 一次塞 3 个类比——稀释洞见密度，选最准的那一个
- 用类比掩盖技术判断——核心数据模型该改还是要改，不是讲个故事就能糊过去

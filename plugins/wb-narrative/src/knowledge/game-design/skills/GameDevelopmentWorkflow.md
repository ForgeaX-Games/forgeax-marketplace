# 游戏开发工作流

## 概述

本文档定义了一个分阶段的游戏开发工作流，遵循"核心优先、测试驱动、迭代完善"的原则。

```
┌─────────────────────────────────────────────────────────────────────────────┐
│                           游戏开发工作流总览                                   │
├─────────────────────────────────────────────────────────────────────────────┤
│                                                                             │
│   Phase 1          Phase 2          Phase 3          Phase 4               │
│   ┌─────────┐      ┌─────────┐      ┌─────────┐      ┌─────────┐          │
│   │ 核心功能 │ ──▶ │ Bug测试  │ ──▶ │ 围绕系统 │ ──▶ │ 完整内容 │          │
│   │   实现   │      │   修复   │      │ 测试数据 │      │ 数据表   │          │
│   └─────────┘      └─────────┘      └─────────┘      └─────────┘          │
│       │                │                │                │                 │

```

## Phase 1: 核心功能实现

**目标**：实现游戏最小可玩版本（MVP），验证核心玩法

### 1.1 确定核心功能清单

在开始开发前，明确列出核心功能：

```markdown
## 核心功能清单模板

### 必须实现 (P0)
- [ ] 功能A - 描述
- [ ] 功能B - 描述
- [ ] 功能C - 描述

### 核心依赖 (P0 所需)
- [ ] 依赖系统X
- [ ] 依赖系统Y
```

**示例（宝可梦类游戏）**：

| 优先级 | 核心功能 | 说明 |
|--------|----------|------|
| P0 | 角色移动 | 四方向移动、碰撞检测 |
| P0 | 战斗系统 | 回合制战斗、伤害计算 |
| P0 | 野生遭遇 | 草丛随机遭遇 |
| P0 | 宝可梦创建 | 创建、属性计算 |
| P0 | 场景管理 | 世界场景、战斗场景切换 |

### 1.2 实现顺序

```
1. 基础引擎层
   ├── 输入系统 (Input)
   ├── 场景管理 (SceneManager)
   └── 渲染循环 (Game Loop)
   
2. 核心游戏逻辑
   ├── 实体系统 (Entity/Pokemon)
   ├── 核心战斗 (Combat)
   └── 状态管理 (GameState)
   
3. 核心交互
   ├── 玩家控制
   ├── 基础UI
   └── 场景切换
```

### 2.3 Bug记录格式

```markdown
## BUG_REPORT.md

### Bug #001
- **状态**: 🔴 Open / 🟡 In Progress / 🟢 Fixed
- **严重程度**: Critical / Major / Minor
- **描述**: 简短描述
- **复现步骤**:
  1. 步骤1
  2. 步骤2
  3. 期望结果 vs 实际结果
- **错误信息**: 控制台错误（如有）
- **修复方案**: 修复描述
- **修复文件**: 涉及的文件列表
```

### 2.4 常见Bug类型

| 类型 | 示例 | 检查方法 |
|------|------|----------|
| 类型错误 | `Cannot read property 'x' of undefined` | TypeScript编译检查 |
| 逻辑错误 | 伤害计算不正确 | 单元测试 + 日志 |
| 状态错误 | 战斗结束后状态未清理 | 流程测试 |
| 边界错误 | 队伍为空时崩溃 | 边界条件测试 |
| 资源错误 | 图片/数据加载失败 | 网络请求检查 |

### 4.3 数据完整性检查

```typescript
// 数据验证脚本示例
function validateGameData() {
  const errors: string[] = [];
  
  // 检查宝可梦数据完整性
  for (const pokemon of allPokemon) {
    if (!pokemon.baseStats) errors.push(`${pokemon.id}: 缺少baseStats`);
    if (!pokemon.types?.length) errors.push(`${pokemon.id}: 缺少types`);
    if (!pokemon.learnset?.length) errors.push(`${pokemon.id}: 缺少learnset`);
  }
  
  // 检查技能引用有效性
  for (const pokemon of allPokemon) {
    for (const move of pokemon.learnset) {
      if (!allMoves.find(m => m.id === move.move)) {
        errors.push(`${pokemon.id}: 引用了不存在的技能 ${move.move}`);
      }
    }
  }
  
  // 检查地图传送点
  for (const map of allMaps) {
    for (const warp of map.warps) {
      if (!allMaps.find(m => m.id === warp.targetMap)) {
        errors.push(`${map.id}: 传送到不存在的地图 ${warp.targetMap}`);
      }
    }
  }
  
  return errors;
}
```

## 工作流执行模板

### 每日开发流程

```
1. 开始工作
   ├── 检查当前Phase
   ├── 查看待办任务
   └── 确认今日目标

2. 开发
   ├── 实现功能/修复Bug
   ├── 编写必要测试
   └── 更新文档

3. 结束工作
   ├── 提交代码
   ├── 更新任务状态
   └── 记录明日计划
```

### 阶段切换检查

```markdown
## 阶段切换确认

从 Phase N 到 Phase N+1:

1. [ ] 当前阶段所有P0任务完成
2. [ ] 当前阶段检查清单全部通过
3. [ ] 无阻塞性问题
4. [ ] 代码已提交并通过编译
5. [ ] 文档已更新
```

## 总结

| 阶段 | 目标 | 数据规模 | 完成标准 |
|------|------|----------|----------|
| Phase 1 | 核心功能 | 最小化（3-5个样本） | 可运行、可战斗 |
| Phase 2 | Bug修复 | 不增加 | 稳定、无阻塞Bug |
| Phase 3 | 围绕系统 | 测试规模（20-30%） | 完整游戏循环 |
| Phase 4 | 完整内容 | 完整（100%） | 发布就绪 |

**关键原则**：
1. 先让核心跑起来，再考虑完善
2. 每个阶段都要稳定后再进入下一阶段
3. 数据随功能逐步扩展，不要过早创建完整数据
4. 持续测试，及时修复Bug

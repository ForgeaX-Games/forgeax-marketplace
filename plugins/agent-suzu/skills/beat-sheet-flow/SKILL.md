# Beat-Sheet Flow

## When to use

- 拿到 Iori 的 pillars.md / spec.md / loop.md，开始写 ux-flow.md / onboarding.md
- 玩家测试反馈"前 30 秒看不懂"——需要回到节拍表回溯哪一拍漏了
- 准备给 cc-coder 写 hud-spec 之前——必须先有节拍才能反推 HUD

**不要**用在：玩法本身还没定（让 Iori 先出柱），或者只是改一个 icon 文字（杀鸡用牛刀）。

## Procedure

1. **画时间轴**：从 0 秒到约 180 秒（前 3 分钟），标四个关键锚点：
   - **0s** 启动 / 第一帧
   - **第一次操作**（玩家第一次按下任何按钮的瞬间）
   - **第一次反馈**（系统对玩家做出的第一次回应）
   - **第一次失败**（玩家第一次受到惩罚 / 被 game over / 被打断）
2. **填四列节拍表**（Markdown 表格）：

   | 时刻 | 玩家动作 | 系统反馈 | UI 变化 |
   |---|---|---|---|
   | 0s | （什么都没按） | logo 淡入 | 黑屏 → logo |
   | 3s | 看到 prompt "按任意键" | bgm 淡入 | "按任意键" 闪烁 |
   | ... | ... | ... | ... |

3. **每行自检三问**：
   - 玩家此刻**为什么**会做这个动作？（动机—不是被命令）
   - 系统反馈是**触觉/视觉/听觉**哪一种？必须具体
   - UI 变化是**新增**还是**修改**？新增的元素之后要列入 hud-spec
4. **反推 HUD**：节拍表里出现的所有 UI 元素，**才**进 hud-spec.md。没出现的不进。
5. **节拍密度判定**：第一分钟内"系统反馈"列必须每 5-8 秒一格，否则玩家会感到"空"。空就回去插一个微反馈。
6. **写明 30s / 60s / 180s 解锁了什么**：onboarding.md 顶上必须有这三个里程碑——什么机制在这一刻第一次出现。

## Examples

- ✅ 三栏节拍：

  | 时刻 | 玩家动作 | 系统反馈 | UI 变化 |
  |---|---|---|---|
  | 12s | 第一次按方向键 | 角色脚步声 + 屏幕轻微抖动 | 体力 bar 微减 |
  | 18s | 撞到第一面墙 | 钝击音效 + 角色撞墙动画 | 出现 "↑ 跳跃" 提示 |

- ✅ "30s 解锁了基础移动 + 跳跃；60s 解锁第一个机制（蹲）；180s 第一次 game over，给重试提示。"
- ❌ "玩家应该会先探索一下场景。"——没时刻、没具体动作、没反馈
- ❌ HUD 里有"魔力 bar" 但节拍表前 3 分钟里玩家从没看过它——删。

## Anti-patterns

- 从 HUD 反推节拍（"我有 X bar 所以玩家这里应该看 X bar"）——颠倒因果
- 节拍只写玩家动作不写系统反馈——玩家会感到"按了没反应"
- 教学行写"按 X 跳跃"——这是命令不是动机；改写"玩家会先因为什么动机想跳"
- 节拍表跨度超过 5 分钟——前 3 分钟最关键，超出范围属于关卡设计不属于 onboarding
- 把节拍密度判定跳过——空场是 onboarding 最大杀手，必须自检
